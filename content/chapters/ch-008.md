# Production Agent Deployment

## Overview

Building an agent locally is one thing. Deploying it to production — where it serves thousands of users concurrently, handles failures gracefully, and costs you money per call — is another challenge entirely. This chapter covers containerization with Docker, deployment to Fly.io, environment management, health monitoring, and graceful error handling.

## Key Concepts

**What Changes in Production?**
- **Concurrency**: Multiple users call your agent simultaneously
- **Cost**: Every API call costs money — budget and monitor carefully
- **Reliability**: Your agent must handle API timeouts, rate limits, and DB failures
- **Security**: API keys must never appear in code or logs
- **Observability**: You must know when something breaks before users tell you

**Deployment Stack for This Project**
| Component | Technology | Purpose |
|-----------|-----------|---------|
| Backend API | Fly.io | FastAPI + asyncpg |
| Database | Neon PostgreSQL | Persistent data |
| Content Storage | Cloudflare R2 | Markdown files |
| Frontend | Vercel | Next.js app (Phase 3) |

## Deep Dive

**Dockerfile for FastAPI**
```dockerfile
FROM python:3.12-slim

WORKDIR /app

# Install dependencies first (layer caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY app/ ./app/
COPY migrations/ ./migrations/

# Non-root user for security
RUN adduser --disabled-password --gecos "" appuser
USER appuser

EXPOSE 8080
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8080"]
```

**Health Check Endpoint**
```python
@app.get("/health")
async def health_check(db=Depends(get_db)):
    """Health check for load balancer and deployment verification."""
    try:
        await db.fetchval("SELECT 1")
        db_status = "ok"
    except Exception:
        db_status = "error"

    return {
        "status": "ok" if db_status == "ok" else "degraded",
        "database": db_status,
        "version": "1.0.0",
        "llm_calls": 0,  # Constitutional guarantee
    }
```

**Graceful Error Handling**
```python
import logging
from fastapi import HTTPException

logger = logging.getLogger(__name__)

async def safe_db_call(db, query: str, *args):
    """Execute a DB query with proper error handling."""
    try:
        return await db.fetch(query, *args)
    except asyncpg.TooManyConnectionsError:
        logger.error("DB connection pool exhausted")
        raise HTTPException(status_code=503, detail="Service temporarily unavailable")
    except asyncpg.PostgresError as e:
        logger.error("Database error: %s", e)
        raise HTTPException(status_code=500, detail="Database error")
```

## Code Example

```python
# Structured logging for production observability
import logging
import json
from datetime import datetime

class JSONFormatter(logging.Formatter):
    """JSON log formatter for structured logging."""
    def format(self, record):
        return json.dumps({
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "module": record.module,
            "function": record.funcName,
        })

# Configure logging in main.py
logging.basicConfig(level=logging.INFO)
for handler in logging.root.handlers:
    handler.setFormatter(JSONFormatter())

# Usage in any service
logger = logging.getLogger(__name__)
logger.info("Chapter accessed", extra={"chapter_id": "ch-001", "user_id": "u-123"})
```

## Summary

- Docker + Fly.io = portable, scalable deployment for FastAPI backends
- Secrets go in Fly.io secret store — never in fly.toml or code
- `/health` endpoint enables automatic health checking and recovery
- Always handle `asyncpg` errors explicitly with proper HTTP status codes
- Structured JSON logging enables production debugging without code changes
- `auto_stop_machines = true` on Fly.io reduces idle costs dramatically
- Run `fly secrets set` for all environment variables before first deploy
